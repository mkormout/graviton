**Change 2 — Add _restart_game() method after _on_player_died() (after line ~399):**
```gdscript
func _restart_game() -> void:
	get_tree().paused = false
	death_screen.visible = false
	_wave_clear_pending = false
	_wave_hud.hide_wave_clear_label()

	for enemy in get_tree().get_nodes_in_group("enemy"):
		enemy.queue_free()

	for child in get_children():
		if child is Item:
			child.queue_free()

	for child in get_children():
		if child is Asteroid:
			child.queue_free()

	var ship := $ShipBFG23
	ship.global_position = Vector2.ZERO
	ship.linear_velocity = Vector2.ZERO
	ship.angular_velocity = 0.0
	ship.health = ship.max_health
	ship.dying = false

	ScoreManager.reset()
	$WaveManager.reset()
	MusicManager.reset()

	spawn_asteroids(100)
	$WaveManager.trigger_wave()
```

Implementation rationale (enforce these, do not deviate):

1. `get_tree().paused = false` is FIRST — unpauses before any queue_free() to prevent silently-deferred deletions (RESEARCH.md Pitfall 3)

2. `death_screen.visible = false` — hides the overlay before cleanup; player does not see the death screen during the reset flash

3. `_wave_clear_pending = false` + `_wave_hud.hide_wave_clear_label()` — mirrors _on_player_died() cleanup; prevents stale wave-clear UI (RESEARCH.md open question 2)

4. Enemy cleanup uses `get_tree().get_nodes_in_group("enemy")` — enemies are always in the "enemy" group (WaveManager._spawn_enemy() calls add_to_group("enemy") at line 94)

5. Item and Asteroid cleanup uses `get_children()` with `is Item` / `is Asteroid` — these classes have no group tag (RESEARCH.md Pitfall 6). Two separate loops required because `get_children()` iterator must not be modified mid-loop — second loop picks up items after enemies from first loop are queued

6. Ship reset (per D-05): position=ZERO, linear_velocity=ZERO, angular_velocity=0.0, health=max_health, dying=false. `ship.dying = false` is MANDATORY — Body.dying is set to true in die() and never auto-reset; without this the ship is unkillable in the next run (RESEARCH.md Pitfall 1)

7. Reset call order (per D-08): ScoreManager.reset() → $WaveManager.reset() → MusicManager.reset()

8. `spawn_asteroids(100)` re-calls the same function used in _ready() with count=100 (same as initial world spawn at game start)

9. `$WaveManager.trigger_wave()` starts Wave 1; WaveManager._current_wave_index was zeroed by reset(), so trigger_wave() spawns index 0 (Wave 1), not a subsequent wave (RESEARCH.md Pitfall 2)

Do NOT call `get_tree().reload_current_scene()` — explicitly forbidden per D-08 and RESEARCH.md.
